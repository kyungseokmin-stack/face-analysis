# 관상 분석 (face-analysis)

고개를 좌우·상하로 돌리며 촬영한 짧은 영상을 기반으로, 고전 관상학 문헌의 개념 체계(삼정·오악·
오관·십이궁·오행 얼굴형)에 대입해 결과를 보여주는 모바일 웹앱. 빌드 도구 없는 정적 사이트이며,
이 저장소 하나만으로 동작한다.

원래 `shilla-search`(신라 상품 검색 앱) 저장소의 `face-analysis/` 하위 폴더로 개발되었으나,
GitHub Pages 배포를 위해 이 별도 public 저장소로 분리했다(`shilla-search`는 private 저장소라
무료 플랜에서는 Pages를 켤 수 없다). 코드 자체는 완전히 독립적이라 분리에 따른 기능 변화는 없다.

## 배포

`main` 브랜치에 푸시되면 `.github/workflows/deploy-pages.yml`이 GitHub Pages로 자동 배포한다
(빌드 단계 없이 저장소 루트를 그대로 올림). 커스텀 도메인은 `CNAME` 파일(`face.kenmin.kr`)로
지정되어 있으며, DNS에서 `face.kenmin.kr`을 `kyungseokmin-stack.github.io`로 CNAME 연결해야
한다.

## 왜 사진 한 장이 아니라 영상인가

MediaPipe FaceLandmarker는 얼굴 표면 468~478개 지점의 3D 좌표(x, y와 카메라 기준 상대 깊이 z)를
프레임마다 추출한다. 이때:

- **깊이(z) 정보는 각도·조명에 따라 단일 프레임에서 잡음이 크다.** 정면·좌회전·우회전·상하 기울임
  등 여러 각도에서 독립적으로 계산한 값(예: 코 돌출도, 삼정 비율)을 평균하면 오차가 줄어든다
  (`js/measurements.js`의 `computeMeasurements`).
- **라이브니스(liveness) 신호가 된다.** "반대 방향으로 돌려주세요" 같은 실시간 지시를 그 순간
  따라야 하므로, 미리 준비된 사진으로 흉내 내기 어렵다.
- 다만 **귀는 이 모델의 랜드마크 토폴로지에 아예 포함되지 않는다.** 브라우저에서 쓸 수 있는
  검증된 귀 전용 랜드마크·얼굴 파싱 모델도 조사해봤지만(BiSeNet 계열 face-parsing, 귀 전용 CNN 등)
  전부 연구용 PyTorch/ONNX 코드일 뿐 브라우저용으로 검증된 패키지가 없고, 직접 변환해도 머리카락에
  가리기 쉬운 부위 특성상 정확도를 장담하기 어렵다. 그래서 "옆모습을 보면 귀 관상까지 정확히
  분석된다"는 과장은 하지 않고, 좌우 회전(turnA/turnB) 촬영 프레임에서 귀 높이 대역을 그대로
  크롭해 참고 사진으로 보여주고(`js/earCrop.js`), 전통적 판단 기준(귓불·크기·위치·각도)을
  체크리스트로 제시해 사용자가 직접 대조하게 한다(`EAR_CHECKLIST`, `js/physiognomyKnowledge.js`).
  자동 판정은 하지 않는다.

## 아키텍처

전부 빌드 도구 없는 바닐라 JS + ES 모듈. 브라우저가 CDN에서 라이브러리/모델을 직접 불러온다.

```
face-analysis/
├── index.html              # 단일 페이지 앱: intro → consent → capture → analyzing → report
├── privacy.html            # 개인정보·보안 안내 (별도 정적 페이지)
├── css/style.css
└── js/
    ├── faceEngine.js        # MediaPipe FaceLandmarker 래퍼, 변환행렬→yaw/pitch/roll 계산
    ├── captureFlow.js        # 정면→좌우회전→상하기울임 5단계 가이드 촬영 상태머신
    ├── measurements.js       # 랜드마크 → 삼정/오악/오관/얼굴형 비율 계산
    ├── physiognomyKnowledge.js # 고전 문헌 기반 해석 규칙(데이터+로직, DOM 비의존)
    ├── reportEngine.js       # measurements + knowledge → 리포트 데이터 구조
    ├── infographic.js        # 정면 사진 위에 실제 랜드마크 윤곽선+콜아웃+측정선(삼정 경계·대칭축·눈코입 캘리퍼, 실측 수치 포함)+리포트 전문을 그려 다운로드 캔버스 생성
    ├── earCrop.js             # 좌우 회전 프레임에서 귀 높이 대역을 크롭 (자동 판정 없음, 참고 사진용)
    ├── sound.js               # Web Audio로 생성하는 단계 완료 알림음(파일 없음, 네트워크 요청 없음)
    └── main.js                # 화면 전환, 카메라 제어, 이벤트 와이어링
```

데이터 흐름: `main.js`가 카메라를 열고 `CaptureFlow`를 시작 → 5단계(front/turnA/turnB/tiltA/tiltB)
동안 매 프레임 `FaceEngine.detectForVideo()`로 자세를 추정, 목표 각도에 일정 시간 머무르면 그 프레임의
랜드마크를 캡처하고 알림음(`playStepDing`/`playAllDone`)을 재생해 화면 문구를 계속 읽지 않아도 다음
동작 시점을 알 수 있게 한다 → `front`/`turnA`/`turnB` 단계에서 비디오 프레임을 캔버스로 스냅샷해
`photoDataUrl`도 함께 저장(정면은 인포그래픽용, 좌우 회전은 귀 참고 사진용) → 5단계 완료 시 카메라
스트림을 즉시 종료 → `computeMeasurements()`로 비율과 콜아웃 좌표(`anchors`) 계산 →
`cropEarBand()`로 turnA/turnB 사진에서 귀 높이 대역을 크롭 → `buildReport()`로 문헌 기반 해석
문구 생성, `buildInfographic()`으로 정면 사진+콜아웃+리포트 전문을 한 장에 그린 인포그래픽 캔버스
생성(사진만 저장해서는 설명을 알 수 없으므로, 다운로드 한 장으로도 보관이 되도록 리포트 텍스트를
이미지 안에 그대로 그려 넣는다) → `main.js`가 리포트 화면 맨 위에 인포그래픽을, 그 아래에 상세
텍스트 카드(귀 섹션에는 크롭된 참고 사진 + 체크리스트 포함)를 렌더링. 인포그래픽 캔버스는 사용자가
"이미지로 다운로드"를 누를 때만 `downloadCanvas()`로 사용자 기기에 저장되고, 그 전까지는(귀 참고
사진 포함) 브라우저 메모리에만 존재한다.

좌우 방향을 미리 "왼쪽"·"오른쪽"으로 못박지 않고 "편한 방향 → 반대 방향"으로 안내하는 이유는
`js/captureFlow.js` 상단 주석 참고 — 셀피 미리보기의 좌우 반전, 기기별 카메라 좌표 관례 차이로
인해 방향 라벨을 고정하면 안내 문구와 실제 동작이 어긋날 위험이 있어, 실제 감지된 회전 방향을
기준으로 삼는 것이 더 안전하다.

## 문헌 근거와 한계

`js/physiognomyKnowledge.js`에 정리된 삼정(三停)·오악(五嶽)·오관(五官)·십이궁(十二宮)·오행 얼굴형은
「마의상법(麻衣相法)」「유장상법(柳莊相法)」「신상전편(神相全編)」 등에서 비롯되어 후대에 통속적으로
널리 인용되는 틀이다. 십이궁은 처음엔 6궁 중 4궁만 실제로 해석을 출력했는데(나머지 2궁은 정의만
해두고 미사용), 부부궁(눈꼬리)·노복궁(턱 끝)도 기존 측정치(cheekBalance·chinWidthRatio)로 채워
6궁 모두 표시하도록 보강했다. 리포트 맨 끝에는 오행·삼정·오악·오관 핵심 문장을 한데 모은
"종합 총평" 섹션도 추가했다(`reportEngine.js`). 다만:

- 유파·문헌마다 세부 명칭·방위 배속에 차이가 있어 여기서는 통설 수준으로 정리했다.
- 수치 임계값(예: "눈썹이 눈보다 15% 이상 길면...")은 문헌의 정성적 서술을 정량 판정으로
  옮기기 위한 근사치이며, 특정 문헌의 정밀 수치를 그대로 옮긴 것이 아니다. 초기 버전은 임계값을
  감으로 잡아 대부분의 평균적인 얼굴이 "무난함" 구간에 몰려 결과가 서로 비슷해 보이는 문제가
  있었다. 이를 보정하기 위해 눈 사이 간격이 얼굴 너비의 약 46%라는 "new golden ratios for facial
  beauty" 연구([ScienceDirect](https://www.sciencedirect.com/science/article/pii/S0042698909005045),
  [PMC](https://pmc.ncbi.nlm.nih.gov/articles/PMC2814183/))처럼 현대 안면 비율 연구를 참고해
  다섯 단계(매우낮음~매우높음)로 재보정했다(`js/physiognomyKnowledge.js`의 `tier5`,
  `CALIBRATION_NOTE`). 그래도 여전히 근사치이며, 전통 문헌이 직접 제시하는 정밀 수치는 아니다.
- 얼굴형(오행) 분류는 초기 버전에서 순차적 조건문 때문에 특정 유형(금형)에 결과가 쏠리는 편향이
  있었다. 5개 유형의 대표 특징 벡터와의 거리를 비교해 가장 가까운 유형을 고르는 방식으로
  바꿔(`classifyFaceShape`) 이 편향을 줄였다.
- **코 너비 측정 버그**: MediaPipe에는 코 전용 랜드마크가 없어 "눈 아래~입술 위, 얼굴 중앙
  ±25%" 좌표 범위를 코로 근사했는데, 실사용자 테스트에서 이 범위가 넓어 볼·팔자주름 부근 점까지
  잡혀 코 너비가 얼굴 너비의 절반 가까이(0.5) 측정되는 문제가 발견됐다(해부학적으로는 20~25%
  수준이어야 한다). x범위를 ±13%로 좁혀 수정했다(`measurements.js`). 눈 간격·눈 크기 임계값도
  실사용자 실측치를 반영해 중심값을 재조정했다.
- 관상학은 현대 과학적으로 검증된 학문이 아니다. 이 앱은 문화·오락 콘텐츠이며, 결과를 성격·운명
  진단이나 채용·의료 등 중요한 판단의 근거로 사용해서는 안 된다. (`DISCLAIMER_TEXT`)
- **길흉 양면 재검토**: 실사용자 테스트에서 "누구를 찍어도 좋은 얘기만 비슷하게 나온다"는 지적을
  받았다. 원인은 각 부위 해석 문구(`*_TEXT` 딕셔너리)를 작성할 때 불편한 내용을 피하려고 모든
  단계(매우낮음~매우높음)를 전부 긍정적으로만 써놓았기 때문이었다 — 실제 고전 문헌은 그렇지 않고,
  부위·정도에 따라 조심스럽거나 흉하게 보는 경우도 함께 전해진다. 이를 바로잡기 위해 눈썹 길이·
  숱, 눈 크기·미간, 코 높이, 입 크기, 턱 너비, 이마 너비 각각의 전통적으로 불리하게 보는 극단
  구간(주로 veryLow/veryHigh)에 웹 검색으로 확인한 고전적 흉상(凶相) 풀이를 추가했다(예: "눈썹이
  눈보다 짧으면 인복이 부족해 힘들 때 도와주는 사람이 없다", "미간이 좁으면 근심 걱정이 많고
  소견이 좁다", "코가 낮으면 활동 범위가 좁아 재물이 잘 모이지 않는다", "입이 작으면 소심하고
  끈기가 부족하다", "턱이 뾰족하면 의지가 약하고 말년에 덕이 부족하다"). 전통적으로 유리하게
  보는 방향은 그대로 긍정적으로 남겨, 대칭적으로 좋은/나쁜 말을 억지로 끼워 맞추기보다 실제
  문헌의 비대칭적인 길흉 구조를 따랐다. `DISCLAIMER_TEXT`에도 "길흉을 함께 담았으며 어느 쪽도
  실제 그 사람의 가치를 판단하는 근거가 아니다"라는 안내를 추가했다.
- **반복 문구 제거(2-에이전트 구현/검토 루프)**: "결과에서 같은 말들이 너무 많이 나온다"는 지적을
  받아, 구현 담당과 검토 담당 두 서브에이전트를 번갈아 돌리며(구현→검토→재구현→재검토) 고쳤다.
  1라운드에서 두 가지 근본 원인을 찾았다 — (1) `noseProminence` 하나의 측정치가 오악(중악)과
  오관(심변관) 두 섹션에 문자열 치환(`.replace('중악(코)','심변관(審辨官)')`)만 한 거의 동일한
  문장으로 노출되고, 종합 총평에서 그 문장을 세 번째로 그대로 재사용하던 문제, (2) "종합 총평"
  섹션 자체가 다른 모든 섹션의 문장을 그대로 복사-붙여넣기 하고 있어 내용의 100%가 리포트
  앞부분과 중복되던 문제. 오악용 별도 해석(`OAK_NOSE_PROMINENCE_TEXT` — 오악 전체 골격 균형
  프레이밍)을 분리하고, 다른 섹션 문장을 재사용하지 않고 새 문장을 조합하는
  `summarizeSynthesis()`를 추가해 해결했다. 검토 에이전트가 여러 합성 프로필로 재검증하는
  과정에서 2라운드 버그를 추가로 찾아냈다 — 종합 총평 템플릿이 "삼정이 고르게 균형 잡힌 삼정"처럼
  단어를 겹쳐 쓰는 문제, 미간(눈 간격) 요약이 veryLow/low 두 단계에서 같은 문구를 쓰던 문제,
  "특별히 튀지 않는 균형 잡힌 상입니다" 상투구가 오관-코와 십이궁 5개 궁(관록·재백·형제·부부·
  노복궁)에 걸쳐 반복되던 문제, 그리고 이 과정에서 우연히 발견된 별개의 정확성 버그
  (`interpretSamjeong`이 삼정 세 구간 중 이상치(1/3)와의 편차가 가장 큰 구간을 고르되, 그 구간이
  실제로는 다른 구간보다 *작은* 경우에도 항상 "발달한 편"이라고 잘못 서술하던 문제 — 편차의
  부호에 따라 발달/압축 6종 텍스트로 분리해 수정). 두 서브에이전트가 서로 코드를 직접 읽고
  독립적으로 재검증하는 과정을 거쳐 최종 승인 후 반영했다.

## 디자인 시스템

"최신 앱 트렌드에 맞게 레이아웃·폰트를 재편해달라"는 요청을 받아, 이번에도 구현/검토
두 서브에이전트를 번갈아 돌려(`css/style.css` 396→660줄) 다시 다듬었다.

- **디자인 토큰화**: 스페이싱(`--space-1`~`--space-8`, 4px 기준), 라운드(`--radius-sm/md/lg/full`),
  타입 스케일(`--text-xs`~`--text-2xl`)을 커스텀 프로퍼티로 도입해 그때그때 정하던 픽셀값을
  대체했다. 기존 색상 변수 이름(`--bg`,`--text`,`--accent` 등)은 그대로 두고 값만 대비를
  재검증해 다듬었으며, `--surface-alt`·`--accent-strong`·`--on-accent`·`--shadow-sm/lg`를
  새로 추가했다.
- **타이포그래피**: 한글 우선 폰트 스택에 Pretendard를 앞세우고, 큰 제목에는 자간을 좁히고
  두께를 올렸다. 앱 워드마크(`.app-title`)에만 `--font-serif`(세리프) 토큰을 적용해 "고전
  문헌 기반"이라는 브랜드 톤을 유지하면서도 본문은 현대적인 산세리프로 남겼다.
- **카드/버튼/진행 인디케이터**: 소개 화면의 특징 목록을 테두리 있는 카드로 바꾸고, 버튼은
  주/보조 위계를 명확히 하고 최소 높이 52px(탭 타깃 44px 기준 이상)을 확보했다. 촬영 화면의
  `#step-dots`는 활성 점이 알약 모양으로 늘어나는 진행 표시로, `#hold-ring`은 이중 도넛 형태로
  바꿨다.
- **접근성/디테일**: 라이트·다크 테마 모두 WCAG 대비 기준을 재검증했고, `env(safe-area-inset-*)`로
  노치 기기 대응, `prefers-reduced-motion` 존중, 스크롤 시 헤더가 고정되는 sticky header를
  추가했다.
- **검토 과정에서 발견해 함께 고친 버그 2건**: (1) 스티키 헤더에 `backdrop-filter`+반투명을 썼다가
  스크롤 시 아래 텍스트가 겹쳐 보이는 문제가 있어 불투명 배경으로 되돌렸다. (2) 에러 화면의 긴
  URL(`#error-detail`)이 카드 밖으로 넘치는, 리디자인 이전부터 있던 잠재 버그를 `overflow-wrap`/
  `word-break`로 함께 고쳤다.
- **인포그래픽 팔레트 동기화**: `js/infographic.js`의 캔버스는 CSS 커스텀 프로퍼티를 읽을 수
  없어 색상값을 하드코딩(`PALETTE`)해 두고 있었는데, 리디자인 후 새 팔레트 값으로 맞춰 갱신했다
  (다운로드되는 인포그래픽 이미지도 새 배색을 따르도록).
- 스크린샷 기반 시각 검증(라이트/다크, 화면별 전후 비교)은 Playwright로 진행했으며, 검토
  에이전트가 스크린샷을 직접 확인하고 코드 diff(필수 id·CSS 변수명 보존 여부)를 대조해 승인했다.

## 보안·개인정보 설계

- 얼굴 인식은 전부 브라우저 안(WASM)에서 실행되며, 영상·이미지·랜드마크 좌표를 서버로 전송하지
  않는다. 페이지가 CDN에서 받아오는 것은 라이브러리 코드와 모델 파일뿐이다.
- 결과/측정치는 메모리에만 존재하고 localStorage·쿠키·서버 어디에도 저장하지 않는다. 예외는 정면·
  좌회전·우회전 사진 세 장(`photoDataUrl`)뿐이며, 인포그래픽·귀 참고 사진 렌더링에만 쓰이고
  인포그래픽을 사용자가 명시적으로 다운로드를 누를 때만 기기에 저장된다(귀 참고 사진 자체는
  현재 화면 표시용으로만 쓰이며 다운로드 대상에는 포함하지 않는다).
- 카메라는 동의 체크박스를 거친 뒤에만 켜지고, 분석 완료 또는 중단 시 즉시 스트림을 정지한다.
- 자세한 내용은 `privacy.html` 참고.

## 로컬 실행

빌드 과정이 없다. 정적 파일 서버로 이 폴더만 서빙하면 된다(카메라 API는 `https` 또는
`localhost`에서만 동작).

```bash
cd face-analysis
python3 -m http.server 8080
# 브라우저에서 http://localhost:8080 접속
```

배포 시에는 반드시 HTTPS로 서빙해야 카메라 권한 요청이 동작한다.

## 알려진 한계

- 조명이 어둡거나 얼굴이 마스크·안경·머리카락에 크게 가려지면 랜드마크 인식 정확도가 떨어진다.
- 좌우 회전 각도 임계값은 18°→30°→45°로 두 차례 올렸는데도 실사용자 스크린샷에서 귀가 계속 안
  보였다. 각도 문제로 보고 계속 임계값만 올렸던 게 잘못된 진단이었고, 실제 원인은 정면(front)
  단계 측정 정밀도를 위해 넣은 "더 가까이 와주세요" 안내(`estimateFaceSizeRatio` 기반 close-up
  힌트)가 `step.key` 구분 없이 turnA/turnB 단계에도 그대로 걸려 있던 것이었다 — 카메라에 바짝
  붙은 채로 고개를 돌리면 웹캠의 좁은 화각(FOV) 때문에 귀가 통째로 프레임 밖으로 밀려나며,
  이건 각도를 아무리 올려도 해결되지 않는 문제다(얼굴 랜드마크는 귀를 추적하지 않으므로
  `faceSizeRatio`는 애초에 귀가 프레임 안에 있는지에 대해 아무 신호도 주지 못한다). 이를 다시
  살펴보라는 지적을 받고 별도 서브에이전트로 코드를 독립 검증한 뒤, close-up 힌트를 `front`
  단계에만 노출하도록 `main.js`를 수정하고, turnA/turnB 안내 문구에 "카메라와 한 걸음 거리를
  두고"를 추가했다(`captureFlow.js`). 회전 각도(45°) 자체는 유지한다 — 문제였던 적이 없었다.
- **위 수정 이후에도 "좌우 얼굴 인식이 잘 안 되고 귀 사진이 여전히 안 나온다"는 재보고를 받고
  찾아낸 실제 근본 원인**: `.capture-stage` 컨테이너는 `aspect-ratio: 3/4`(세로형)로 고정돼
  있는데, `getUserMedia`가 요청하는 해상도(960×720)는 4:3(가로형)이다. `#camera-video`에
  `object-fit: cover`를 쓰고 있어 이 두 비율이 다르면 미리보기 가로 폭의 약 44%가 화면 밖으로
  잘려나가 사용자에게는 실제보다 훨씬 확대(줌)된 것처럼 보인다. 이 잘린 미리보기를 기준으로
  귀가 보일 때까지 고개를 돌리면, 실제로는 필요한 것보다 훨씬 큰 각도로 돌리게 되어 MediaPipe의
  랜드마크 추적 자체가 끊기고(→"얼굴 인식을 잘 못한다"), 추적이 끊기니 회전 단계의 목표 각도를
  일정 시간 유지하지 못해 캡처도 끝내 발생하지 않는다(→"귀 사진이 안 나온다"). 즉 이전의 두
  가지 수정(각도 상향, close-up 힌트 범위 제한)은 모두 진짜 원인이 아니었고, 이 비율 불일치가
  처음부터의 근본 원인이었다. 카메라 기종마다 실제 스트림의 가로세로 비율이 다를 수 있어
  하드코딩된 값으로는 근본적으로 고칠 수 없으므로, `main.js`에서 `videoEl`의 `loadedmetadata`
  시점(및 재생 직후)에 `.capture-stage`의 `aspect-ratio`를 실제 스트림 값(`videoWidth/videoHeight`)
  으로 덮어써서, 어떤 기기에서도 미리보기가 항상 캡처되는 원본 프레임 그대로(잘림 없이)
  보이도록 했다(`syncCaptureStageAspectRatio()`). Playwright로 확인한 결과 컨테이너와 비디오의
  렌더링 크기가 정확히 일치하게 되었다(예: 960×720 스트림 → 348.75×261.56px, 두 값 동일).
- 오악 해석의 좌/우 표기는 화면에 보이는 방향 기준이며, 카메라 좌우반전 여부에 따라 해부학적
  좌/우와 다를 수 있다.
- 얼굴이 화면에서 너무 작으면(멀리서 촬영) 랜드마크 정밀도가 떨어져 측정이 부정확해질 수 있어,
  정면(front) 단계에서만 전체 랜드마크의 가로 폭이 프레임의 32% 미만이면 "더 가까이" 안내를
  띄운다(`captureFlow.js`의 `estimateFaceSizeRatio`, `main.js`에서 `front` 단계로 한정). 카메라
  해상도도 640×480→960×720으로 높였다.
